0 · 系统总览与设计目标

本系统是一个单次运行（或定时守护）的命令行管线：每次运行从数十个数据源并发抓取原始资讯， 经过多层客观过滤 + 打分把噪声压下去，再交给 LLM 按用户身份逐分类挑选， 最后生成一份结构化的中文 HTML 日报并推送。

核心设计原则

  • 身份驱动：所有筛选维度都围绕一个虚拟读者 —— 「为大规模 AI 训练/推理集群做存储栈选型、容量/带宽规划、架构演进、TCO 测算、跨厂商对标」的分析师。
  • 客观先行、LLM 收尾：用规则/打分把 100+ 候选压到可控规模（省 token、保判断质量），LLM 只在高信噪比的候选池里做「重要性」终选。
  • 宁缺毋滥：每个分类候选不够格就少选甚至空列表，不为凑数放低质内容。
  • 反幻觉：摘要优先用 RSS 原文；LLM 生成的摘要要二次自检；标题里的数字/公司主体必须能在原文中找到，否则强制重写。
  • 绝不伪造时间戳：拿不到发布日期的条目一律视为「时间未知」，在限定时间窗的运行里宁可丢弃。

1 · 数据源

数据源集中配置在 config/sources.yaml，每个源声明 type、category 与 enabled 开关。 共三种采集类型：

type: rss 直接拉

固定 RSS/Atom URL，整体抓回后再做关键词匹配。涵盖官方博客、行业媒体、arXiv、GitHub Trending、中文媒体、意见领袖。

type: keyword_search 逐关键词

把每个关键词（+ search_as 独特别名）替换进 URL 的 {q}，每个词发一次请求。当前主力是 Google News RSS 搜索。

type: tavily 搜索 API

走 Tavily Search API，接近浏览器搜索覆盖度（含国内/官方/非主流站），弥补 Google News 盲区。免费档约 1000 次/月，需 TAVILY_API_KEY。

微信公众号 需中继

公众号无官方 RSS，URL 含 {base} 占位，需自建 wechat2rss / RSSHub 并设 RSSHUB_BASE，未配置时整批静默跳过。

1.2 关键词与主题配置

文件内容
config/keywords.yaml追踪关键词清单。每个词可带 aliases（别名）、official_domains（权威域名）、case_sensitive、match_aliases_only、search_as（搜索专用独特别名）。覆盖存储厂商、文件系统、GPU IO、互联协议、KV cache/checkpoint、向量库/数据平台、云厂商、大模型、训练推理框架、AI 芯片、基准评测、行业标准等约 100+ 词。

config/themes.yaml语义主题清单（供主题门召回）：KV cache、向量/图数据库、并行文件系统、checkpoint、GPU IO。每个主题含 keyword（归入哪个既有关键词）、definition（喂给 LLM 的定义）、vocab（本地粗筛词表）。
config/llm.yamlLLM 配置：当前用 MiniMax-M2.7（OpenAI 兼容接口），top_k_per_category=6、max_candidates_per_category=55。

2 · 抓取层（提取）

采集核心在 collectors/rss.py 的 fetch_all()。所有源（含每个搜索关键词的每次查询） 被展开成扁平的 task 列表，每个 task = 一次 HTTP 请求，全部丢进线程池并发执行。

2.1 任务展开（_build_tasks）

  • type=rss：1 个源 = 1 个 task。
  • type=keyword_search：对每个关键词，搜 name + 其 search_as 列出的独特别名（如华为的「Tao 定律」），各发一次 → N 个 task。
  • type=tavily：同理对每个关键词 + 别名各查一次，需 API key，未配置则跳过。

2.2 HTTP 抓取层（httpx，替代 feedparser 裸 urllib）

feedparser 内置的 urllib 没超时、没重试、没 UA、不限并发，对 news.google.com / hnrss.org 这类单 host 高并发源会大面积 SSL EOF / 502。因此自建 http_get_bytes()：

  • 浏览器 User-Agent + 20s 超时（NEWS_HTTP_TIMEOUT）+ 跟随 301/302 跳转。
  • 每主机并发限流：同一 host 最多 4 个并发（NEWS_PER_HOST_LIMIT），用 host 级信号量。
  • 分级重试：4xx（除 429）视为永久失败立即放弃；429 / 5xx / 网络错误 / 超时视为瞬时，指数退避重试 3 次。退避在释放信号量之后进行，不占着名额睡觉。
  • 线程池默认 16 worker（NEWS_FETCH_WORKERS），as_completed 流式 yield。

2.3 统一 item schema

无论来自哪种源，每条原始数据都被归一化成同一结构：

{
  "id":       sha1(url|title)[:16],   # 去重主键
  "source":   "源名（搜索源带 (关键词) 后缀）",
  "category": "official | storage | paper | news | influencer ...",
  "title":    "...",
  "url":      "...",
  "summary":  "原始 RSS 摘要 / Tavily content（可能含 HTML）",
  "published":"UTC ISO 字符串，拿不到则为空串（绝不伪造 now）",
  "signals":  {points, comments, tavily_score, ...},   # 各源能给的客观信号
  "matched_keywords": []              # 由 normalizer 填
}
发布时间解析：依次尝试 published_parsed / updated_parsed / published / updated / pubDate / date， 全失败返回空串。空串在后续时间窗里被当作「未知发布时间」处理 —— 这是「绝不伪造时间戳」原则的落点。

3 · 归一化：去重 / 时窗 / 关键词匹配

抓回的原始条目进入 core/normalizer.py 的 normalize()，依次执行下列步骤。

去重（dedupe）
按 id（url+title 的 sha1）去掉重复条目。
时间窗过滤（after / since）
只保留 published ≥ since 的条目。无可解析日期 → 直接丢弃（限定窗口里宁缺勿滥，避免无法证明在窗口内的旧闻漏进来）。
关键词匹配（match_keywords）
在 title + summary 上匹配关键词及别名，命中则写入 matched_keywords。
质量过滤（is_low_quality）
命中关键词后再过一道质量闸（见第 4 节）。
语义主题门（theme_gate）
未命中关键词、但来自具名媒体源的条目按内容召回（见第 5 节）。
delta 计算（SQLite）
从历史快照算 delta_24h（热度增量），并把当前信号 upsert 落库。
跨源聚合 + 事件折叠
统计同事件被几家媒体报道、折叠近重复转载（见第 7 节）。
watchlist 标记 + 打分
标记重点关注条目，给每条算客观分 score（见第 6 节）。

3.1 时间窗逻辑（compute_since）

模式窗口起点
默认过去固定 24h（不再用 last_run_at 收窄，避免高频运行时各分类凑不齐数据）
--window-hours N强制过去 N 小时，且不更新 last_run_at
--all不限时间窗（since=None）

内部一律存 UTC，显示时转本地时区（fmt_local）。

3.2 关键词匹配规则

  • 英文：单词边界正则 (?<![A-Za-z0-9])term(?![A-Za-z0-9])，避免子串误命中。
  • 中文：直接 substring 匹配。
  • case_sensitive：公司/产品名建议开 true 避免误伤（如 WEKA、CXL）。
  • match_aliases_only：只用别名匹配，name 仅作显示。用于 Ray、Dynamo、MoE、Triton、checkpoint 这类裸词误命中率极高的关键词 —— 例如 MoE 会撞 Ministry of Education，Ray 会撞 Ray-Ban。
  • 意见领袖源例外：category=influencer 的条目绕过关键词匹配（大神发言正文未必含存储词，但本身即精选），只剔除 RT/回复，用源名里的人名作为 matched_keyword。

4 · 质量过滤

命中关键词后并不直接通过，还要过 is_low_quality() 这道闸，剔除社区闲聊/求助/营销/股评，避免污染候选池。判定维度：

维度规则示例
域名黑名单内容农场 / 股评聚合站直接拦simplywall.st、fool.com、seekingalpha.com、benzinga.com、zacks.com 等
股价/估值模板标题命中模板词即拦（即使域名不在黑名单）"price target"、"should you buy"、"stock surges"、"估值分析"、"目标价"
社区疑问句来自社区源 + 标题以 ? 结尾Reddit/HN 上的个人提问
闲聊/求助前缀标题以特定前缀开头"Ask HN"、"how to"、"why is"、"anyone else"、"请问"、"求助"、"求推荐"
营销腔个人/标题党片段"I built"、"you won't believe"、"must read"
过短无信息社区源 + 标题 <12 字符且全英文Reddit 上 1-2 词的帖
域名黑名单会被检查两次：normalize 阶段一次；以及 LLM 选完、Google News URL 解码出真实域名后再过一次（is_blacklisted_url）—— 因为 Google News 链接在解码前看不到背后的真实站点。

5 · 语义主题门（内容召回）

纯字符串精确匹配有召回缺口：下一个 Reasonix 类 KV cache 方案、Lakebase 类向量库，若标题里不带已知关键词就会被丢。core/theme_gate.py 解决这个问题 —— 从内容出发识别主题，而非枚举每个产品名。

5.1 成本控制三道闸

  1. 只对具名媒体源跑：仅 source_tier ≥ 1.0（具名 RSS / 一手博客 / 中文 AI 媒体）的未命中条目才进，Google News / Tavily / GitHub Trending / 意见领袖被排除。
  2. 本地词表粗筛：先用所有主题 vocab 的并集做小写子串匹配，命中的才送 LLM（一次合并调用）。
  3. 上限截断：单次最多 80 条，超出按「源质量 + 新近度」排序取前 N，其余明确告知本轮跳过、不静默丢。

5.2 判定与产出

LLM 对每条判断「是否确实在讲某个关注主题」（只是顺带提一句不算），从严判定，拿不准填 none。命中则：

  • 以该主题对应的既有 keyword 重新纳入候选池（正常进入后续分类/排序）；
  • 抽取「让它相关的最核心实体名」（如某新产品名）。
关键词自动沉淀：召回时把核心实体名计数落盘到 reports/theme_finds.json。某新名字累计出现 ≥ 3 次且仍不在 keywords.yaml 里，就打印「建议固化为关键词」—— 让长期反复出现的新名字从「每次花 LLM 的语义门」沉淀为「零成本精确匹配的关键词」。

6 · 客观打分

compute_score() 聚合所有源能给出的客观信号，值域大致 0~2.0。这个分数用于进入 LLM 前的预排序（不直接决定最终入选）。

score =  0.25·log10(points+1) + 0.08·log10(comments+1)   # HN 绝对热度*
       + 0.25·log10(delta_24h+1)                          # 24h 热度增量*
       + 0.20·log2(cross_source_count+1)                  # 多源同时报道（最强 cross-check）
       + 0.15·tavily_score                                # Tavily 自带相关性分 (0~1)
       + 0.07·recency_boost                               # 越新越靠前
       + 0.30·source_tier                                 # 源质量分级

* HN 已禁用，points/comments/delta 多数为 0；当前主力信号是 cross_source_count、source_tier、recency。

6.1 各项说明

项含义 / 取值
log 压缩把 HN points、cross_source 这类离散数量压到可比尺度；cross 用 log2（增长快），被 5 家媒体覆盖的事件权重接近 HN 1000 分。
source_tier源质量分级 0~1：具名 RSS = 1.0（官方/垂直/中文 AI 媒体）>GitHub Trending = 0.6>Google News / Tavily = 0.3。搜索兜底是内容农场/标题党/近重复的主来源，整体降权。
recency_boost0~1，在采集周期内距今越近越接近 1。按采集周期归一化（周报等 >24h 窗口里靠后的天也能拿到新近度信号）。
cross_source_count同事件被多少家不同媒体报道（见第 7 节）。
tavily_scoreTavily 自带的 0~1 相关性分，作为 Tavily-only 条目的兜底信号，避免其客观分恒为 0。
源分级是后期引入的关键修正：HN 禁用后客观分几乎全平，候选池失去有效预排序、全靠 LLM 在噪声里硬挑。 加 source_tier 后把「可信一手/媒体源」和「搜索兜底」拉开档次。

7 · 跨源聚合与事件折叠

7.1 同事件聚类

Google News 标题形如 "Headline - Outlet"，同一事件的多个转载后缀不同。系统用剥离媒体名后缀 + 归一化的标题作聚合 key（_event_key：去 "- 媒体名" → 小写 → 去标点 → 折叠空白），让同一事件的多个版本真正归并。

7.2 两个产出

  • compute_cross_source_count()：给每条写入 cross_source_count（被几家媒体报道）和 cross_sources（媒体名列表，用于报告里「📎 来源：A / B / C」多源标注）。
  • collapse_events()：同一事件的多个转载只留一条代表，去除候选池里的近重复、避免稀释 LLM 判断。代表条选取优先级：中文来源 > 源质量 > 客观分，并自带完整媒体名单。

8 · LLM 分类排序

候选池进入 core/ranker.py 的 rank_by_category()，按 7 个分类各自独立做 LLM 排序。

8.1 七个分类（用户视角）

存储/基建产品

存储厂商/SSD/互联协议/GPU 数据通路 的产品级动态

模型/框架/算法

大模型 release / 训练推理框架 / 推理优化算法

基准评测

MLPerf / IO500 / SPECstorage 等评测结果（选型硬通货）

学术论文

arXiv / 顶会 / 期刊原始论文

政策导向

出口管制 / 监管 / 法规 / 行业标准联盟

github趋势

github.com 的项目动态

AI意见领袖

Twitter/X 大神一手发言（单独 rubric）

8.2 启发式硬分桶（heuristic_category）

每条候选先被规则硬分到某一桶，优先级从高到低（前者命中即归类）：

  1. 意见领袖源 → AI意见领袖
  2. 命中基准关键词 / 标题摘要含 MLPerf·IO500·SPECstorage → 基准评测
  3. arXiv / 论文域名（openreview、usenix、VLDB…）→ 学术论文
  4. github.com 域名 / GitHub Trending 源 → github趋势
  5. 含政策信号词（export control、出口管制、网信办…）或 .gov 域名 → 政策导向（早于官方源判断，防止「工信部发布…」被误判为公司新闻）
  6. 命中关键词所属类别（存储集合 vs 模型集合，存储优先）
  7. 官方源按源名启发式（vLLM/HF/OpenAI… → 模型；其余 → 存储）
  8. 兜底 → 存储/基建产品

8.3 候选池排序 + token 自适应粗筛

每个桶单独处理：

  1. 排序键（降序）：watchlisted › 严格官方源（仅厂商 official_domains）› 任意官方源（含 github/arxiv）› 客观分。
  2. 质量头 + 多样性尾：取 max_pool（默认 55，学术论文 80）条进 LLM。前半按上面排序取最强条目（保质量），后半按关键词轮转（round-robin）补主题覆盖 —— 避免某个高产官方源（一天十几篇 NVIDIA 博客）刷屏，把只来自单源的真新闻（华为 Tao 定律、KV file system）挤出候选池。
  3. token 预算二分粗筛：估算 prompt token，若超预算（默认 20000）：先尝试压缩摘要（compact）；仍超则二分搜索保留条数，用「官方一手保底 ~1/4 + 关键词轮转」策略压到预算内，并打印丢了多少。

8.4 LLM 终选 prompt

每个桶用一段 prompt 调一次 LLM（JSON 输出），核心包含：

  • 身份 rubric：7 个信息价值维度（存储/AI 基建相关性、性能数字、经济数字、跨厂商对标、首发性、决策影响范围、可操作性、时效性），明确「带增量 vs 不带增量」的取舍清单。
  • 分类定义：每个桶有专属 category_def 圈定范围。
  • 挑选规则：宁缺毋滥；【重点关注】条目只要有实质进展就优先纳入；官方源默认保留；同事件多版本只留官方版。
  • 产出：每条给 display_title（≤30 字中文新闻式短标题）+ summary（250-500 字中文摘要，强制「忠实性 > 数字密度 > 字数」，禁止编造数字/填充句/评价词/推测词）。
  • 意见领袖桶用单独 prompt：评判「观点影响力/信息量」，不要求存储相关、不强制数字结构。
Prompt 热更新：rubric 与分类定义可被 config/prompt_override.yaml 逐字段覆盖（界面可改），每次运行重读，无需重启。

8.5 重点关注保底

被 watchlist 命中的最佳在窗新闻，若 LLM 因「宁缺毋滥」漏选，会被强制补回各自分类（每个词补 1 条最高分），再单独调一次 LLM 给它补中文标题 + 摘要。保证「只要当周有该跟踪项的新闻就一定展示」。

9 · 摘要 / 自检 / 翻译 / 标题核查

LLM 选完后，对选中条目做一系列后处理（均在 main.py run_once 中串联）：

摘要策略选择
原 RSS 摘要可用（clean_rss_summary 通过：非元数据/占位符、≥60 字、非促销噪声）→ 直接用原文（summary_source=rss）；不可用 → 用 LLM 生成的摘要（summary_source=llm）。意见领袖发言例外：始终用 LLM 中文转述（含「为何值得知道」），不回退英文原推。
Google News URL 解码
decode_items_inplace：把 base64 加密的中转链接还原成真实文章 URL（只对最终选中条目调用，省请求）。
解码后二次黑名单过滤
解码暴露真实域名后，再过一遍域名黑名单，丢掉命中的 picks。
分类内标题去重
同分类里标题近重复的，只保留客观分最高的一条。
摘要二次自检（verify_all）
仅对 LLM 生成的摘要：核查是否含原文未提及的推测词（可能/或将/有望）、评论词（重大意义/标志着）、编造的细节/数字；有问题则严格基于原文重写。
翻译（translate_all）
RSS 原文摘要非中文 → 全量翻译；LLM 摘要残留韩/日/阿拉伯等非英非中文字 → 翻译清洗。保留专有名词/数字原样。
标题反幻觉核查（verify_titles）
检查 display_title 里的数字和公司/产品主体是否能在原标题或原 RSS 摘要中找到（关键：校验只基于原文，不含 LLM 自己生成的 summary，避免自我印证幻觉）。查不到 → LLM 重写或回退翻译后的原标题。

10 · 综合层（今日概览）

core/digest.py 在所有选中条目上做跨条目综合，一次合并 LLM 调用，生成：

  • 今日概览（overview）：3-5 句把当天信号串成一个判断，必须引用真实出现的事件，禁止编造。
  • 趋势分析（trends） + 规划建议（advice）：可选，由 digest_analysis 开关控制。当前默认关闭 —— 仅凭局部信息难给出高价值建议，只保留「今日概览」，待信息面扩展后再开。

读者定位与排序层一致（资深 AI 存储技术规划分析师），趋势/建议都落在选型、采购、容量/带宽规划、架构演进、TCO 决策层面。失败时优雅降级为空，报告照常出。

11 · 报告输出与推送

  • HTML 日报：reporter/render.py 用 Jinja2 模板（templates/daily.html）按 7 个分类渲染，含今日概览、每条的中文标题/摘要/来源/多源标注/发布时间/原文链接。落盘到 reports/YYYY-MM-DD-HHMM.html。
  • 推送：reporter/notify.py 按 config/notify.yaml 推送（企业微信等渠道）。
  • 控制台输出：分类打印选中条目，含官方/分类徽标、摘要来源标签（rss/llm + 自检/翻译标记）。
  • Token 用量报告：按调用类型分桶统计 LLM 输入/输出 token。
  • 调试模式（--debug）：core/debug_trace.py 记录每一步筛选前后内容与丢弃原因，落盘为可读 HTML + 完整 JSON 到 reports/debug/。

11.1 历史存储（SQLite）

history.sqlite 存三张表：items（元数据 + 首次见到时间）、signal_history（每次抓取的信号快照）、meta（如 last_run_at）。 delta_24h = 当前 points − 24h 前最近一次快照的 points；24h 内首见则 delta=0（靠 recency 兜底）。

11.2 运行方式

python main.py                       # 单次跑，过去 24h 窗口
python main.py --window-hours 24     # 强制固定 24h 窗口
python main.py --keywords NVIDIA,vLLM  # CLI 临时覆盖关键词
python main.py --no-llm --top 30     # 跳过 LLM，按客观分出列表
python main.py --debug               # 记录每步筛选前后内容
python main.py --daemon --at 09:00   # 每天 09:00 定时跑
